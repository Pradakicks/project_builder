# PRD — Project Builder

> What features exist and their requirements. Every code change must trace back to a requirement here. Source of truth: `README.md`, `docs/architecture.md`, `docs/next-steps.md`.

## Overview

Project Builder is a desktop application (Tauri) for orchestrating AI-agent-driven software development. It sits one layer above coding agents (Claude Code, Cursor, Codex) and lets non-technical users manage large, long-lived software projects as a visual diagram.

Users compose a project as a graph of **pieces** (components/scopes) connected by **relationships**. A **CTO agent** holds the full project context and makes high-level decisions. A **Leader agent** turns the diagram into structured work plans. **Piece agents** (one per component) execute tasks via external coding tools or a built-in LLM. The system supports three autonomy modes (manual / gated / autopilot) with configurable review gates.

## Target user

Non-engineers who want to build and maintain real software projects without writing code themselves, but who want oversight at meaningful decision points.

## Core concepts

- **Project** — a git repo with metadata, pieces, connections, runtime spec, and history.
- **Piece** — a component/module with name, type, interfaces, constraints, agent config, and a phase (design/plan/implement/integrate/runtime).
- **Connection** — a labeled relationship between two pieces with constraints.
- **Work plan** — a structured, ordered list of tasks generated by the Leader agent from the current diagram.
- **Goal run** — one execution of a task by a piece agent, with streamed output, git branch, and result.
- **CTO decision** — an audited action proposed by the CTO agent (create/update/connect/etc.), gated by a review policy.
- **Runtime spec** — install/run/readiness/verify commands for the project's built app.

## Features (shipped)

### F1 — Project lifecycle
- Create / rename / delete / list projects
- Per-project working directory (git repo, initial commit)
- Save/load via native file dialogs

### F2 — Visual editor (canvas)
- Drag to create pieces; draw edges to connect them
- Per-piece editor: name, type, responsibilities, interfaces, constraints, agent config
- Piece nodes show phase pill + execution status indicator
- Persisted layout (positions, viewport)

### F3 — CTO agent
- Chat panel with streaming responses
- Reads full diagram + history as context
- Proposes actions (create/update/connect/delete pieces)
- Review-gated execution per phase-control policy (manual / gated / autonomous)
- Audit log of every decision with execution + rollback metadata

### F4 — Leader agent + work plans
- Generates structured JSON work plan from current diagram
- Plan summary + collapsible task cards (priority color-coded)
- "Run ▶" per task or "Run All ▶" sequential execution
- Inline streaming agent output in each task card
- Inline failure feedback textarea + retry

### F5 — Piece agent execution
- Two backends: built-in LLM (Anthropic / OpenAI-compatible) and external CLI (Claude Code, Codex)
- Per-task git branch, auto-commit on success
- Post-run validation hook
- Token tracking per agent run

### F6 — Branch merge + integration review
- Merge piece branches back to main
- Conflict handling: manual, AI-assisted, auto
- Integration review agent checks cross-piece API/data/config consistency, streamed as markdown

### F7 — Runtime
- Auto-detect install/run/verify commands (static patterns + LLM agent fallback)
- Manual override via runtime spec
- Start/stop subprocess from UI; stream output
- Embedded app URL with readiness probe
- Verify command + goal-run completion

### F8 — Settings
- API keys stored in OS keychain (macOS Keychain / Windows Credential Manager / Linux Secret Service)
- LLM provider/model config
- Per-project runtime spec

### F9 — Agents panel
- All pieces grouped by execution state (idle / running / success / failure / validation-failed)

### F10 — Live documentation enforcement
- Agents update `runtime.json`, design docs, and `docs/next-steps.md` alongside code changes
- See `AGENTS.md` Live Documentation Philosophy section

## Non-goals

- Replacing IDEs or coding agents (Project Builder orchestrates them, not replaces them)
- Hosting / deployment / CI (out of scope)
- Multi-user real-time collaboration
